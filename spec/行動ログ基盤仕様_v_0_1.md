## 1. 概要

- PC上の行動を自動で記録し、あとから主に Web アプリ（PWA）から見返せる個人向け行動ログ・日記システムを定義する。
- 目的は「その日何をしていたか」「最近何に時間を使っているか」を、単なる生ログではなく、読み返せる記録として残すこと。
- 本機能は監視ツールではなく、ユーザー自身の記憶補助、振り返り支援、行動の再発見を目的とする。
- 時刻の基準は原則 JST（UTC+9）とし、表示、保存、日次区切り、週次区切り、検索条件、要約生成の基準日もすべて JST とする。

## 2. 位置づけ

- 本仕様は既存の `quests`, `completions`, `skills` とは別レイヤーの `activity_logs` 基盤を定義する。
- `quests` / `completions` は「成長として扱う記録」、`activity_logs` は「その日に何をしていたかを思い出す記録」として役割を分ける。
- 同じ行動が両方に関係することはある。たとえば Chrome 拡張の閲覧行動は、既存の閲覧クエストにもなりうるし、行動ログにも残る。
- 行動ログは、既存の週次ふりかえりを置き換えない。既存の週次ふりかえりは成長記録中心のままとし、行動ログは別系統の `週次行動レビュー` で扱う。
- リリィの会話、検索、将来のおすすめ生成において、`activity_logs` は重要な入力情報になる。

## 3. 体験目標

- ユーザーが頑張って入力しなくても記録が残る。
- 生ログのままではなく、あとで読める単位にまとまっている。
- `検索して思い出せる` と `流れで読み返せる` の両方を満たす。
- 日記ほど重くなく、トラッカーほど冷たくない。
- 端末が変わっても PWA から同じ履歴を見られる。
- 便利さより先に、止められる、隠せる、消せるという安心感を持てる。

## 4. プラットフォーム構成

### 4-1. システム全体像

- 各 PC に `記録エージェント` を置き、ローカルで行動イベントを取得する。
- v0.1 の Windows 実装では、`lily_desktop` を記録エージェントの正式ホストとし、収集専用の別デスクトップアプリは設けない。
- バックエンドは `RawEvent`, `ActivitySession`, `DailyActivityLog`, `WeeklyActivityReview`, `SituationLog` などを保存する。
- AI 処理層はイベントを整理し、タイトル付け、分類、要約、検索補助、未完了事項抽出を行う。
- 閲覧 UI は `Web アプリ（PWA）` を正式クライアントとする。
- `デスクトップアプリ` は記録収集とリリィ連携のハブとして扱い、閲覧 UI の主役にはしない。
- 将来的に複数 PC やスマホから同一アカウントで同じ履歴を PWA から参照できるようにする。

### 4-2. クライアントごとの役割

- Web アプリ（PWA）は `じっくり見返す`, `検索する`, `週次レビューする`, `手動補正する`, `手動メモを残す` 用途を主とする。
- デスクトップアプリは `記録収集`, `一時停止`, `除外設定`, `リリィ連携` のハブとして機能する。
- v0.1 の Windows 実装では、`lily_desktop` が Desktop Agent のホスト役も兼ねる。
- `lily_desktop` 内部には、UI / 会話処理から責務分離した `Activity Capture Service` を置く。
- `Activity Capture Service` は `アクティブウィンドウ取得`, `アイドル判定`, `Chrome 拡張や他ローカルクライアントからのイベント受信`, `プライバシールール適用`, `ローカル一時保存`, `バックエンド同期への受け渡し` を担当する。
- `Activity Capture Service` の実装形態は、同一プロセス内モジュールでも子プロセスでもよい。v0.1 では同一リポジトリ・同一アプリ配布物の内部コンポーネントとして実装し、将来的なサブプロセス化を妨げない責務分離を行う。
- `Activity Capture Service` に障害が起きても、可能な限り `lily_desktop` の UI や会話機能は継続し、収集機能のみを停止・再起動できる構造を目指す。
- v0.1 の `Activity Capture Service` は `RawEvent` を `lily_desktop/logs/action_logs/raw_events/YYYY-MM-DD.jsonl` へ JSONL で durable spool し、`lily_desktop/logs/action_logs/sync_state.json` に日別ファイルごとの `ackedLines` を保存する。
- `RawEvent` の同期は `ActionLogSyncRequested` を起点に行い、起動時 1 回と `activity_capture.sync_interval_seconds` ごとの定期同期で `/action-log/raw-events` へ best-effort 送信する。v0.1 の既定値は `30` 秒とする。
- `activity_capture.enabled` の既定値は `true`、`activity_capture.initial_state` の既定値は `active`、`activity_capture.poll_interval_seconds` の既定値は `2`、`activity_capture.sync_interval_seconds` の既定値は `30` とする。
- v0.1 では、閲覧専用のネイティブアプリは想定しない。

### 4-3. 同期方針

- 基本方針はローカルファーストとし、PC 側で先に記録してからクラウドへ同期する。
- オフライン中はローカルへ一時保存し、復帰後にベストエフォートで再送する。
- ローカル一時保存キューや再送管理は `Activity Capture Service` の責務とする。
- 閲覧クライアントは必要な範囲をローカルキャッシュしてよいが、正本はバックエンドに置く。

## 5. 記録対象と収集ルール

### 5-1. 収集対象

- アクティブアプリ名
- ウィンドウタイトル
- ブラウザの URL, ドメイン, ページタイトル
- プロジェクト名、ファイル名、作業対象の識別子
- カレンダー予定
- 手動メモ
- 離席 / アイドル状態

### 5-2. 収集しないもの

- キー入力本文
- クリップボード本文
- パスワードや認証コード
- メール本文やチャット本文の全文
- スクリーンショット画像の長期保存
- ユーザーが明示的に除外したアプリ、ドメイン、画面

### 5-3. イベント生成ルール

- 前面アプリまたはウィンドウタイトルが変わった時点で `RawEvent` を生成する。
- 同じ前面コンテキストが継続している間は 30 秒ごとに heartbeat を持ってよい。
- Chrome 拡張や他ローカルクライアントからのイベントは、Local HTTP Bridge などの localhost IPC を通じて `Activity Capture Service` に渡し、Desktop 側イベントと同じ `RawEvent` 系列へ正規化する。
- `source=chrome_extension` の `browser_page_changed` / `heartbeat` では、必要に応じて `metadata.elapsedSeconds` などのブラウザ滞在時間情報を持ってよい。
- `Activity Capture Service` は永続化や同期の前にプライバシールールを適用し、除外対象イベントを破棄またはマスクする。
- ブラウザは `domain` と `title` の両方を持つ。URL 全文を保存するか、ドメインだけにするかはプライバシー設定で切り替える。
- ブラウザのプライベート / シークレットモードは記録対象外とする。
- ファイル連携が使えない環境では `projectName`, `fileName` は `null` を許容する。
- ユーザー操作が 5 分以上ない場合は `idle_started` とみなし、復帰時に `idle_ended` を記録する。
- カレンダー予定は「行動そのもの」ではなく「文脈補完情報」として扱う。

### 5-4. ブラウザ時間計測の住み分け

- ブラウザ内のページ単位・タブ単位の滞在時間の正本は Chrome 拡張とする。
- Chrome 拡張は、アクティブタブ切替、URL 変更、ウィンドウ blur/focus、flush をもとにブラウザ滞在時間を計測し、その結果を `閲覧クエスト` 系集計と行動ログ基盤の両方で利用してよい。
- `Activity Capture Service` は Chrome 拡張由来のブラウザイベントを受け取り、他アプリの出来事と合わせて `RawEvent` / `ActivitySession` に統合する。
- `Activity Capture Service` は、Chrome が前面だった時間だけからページ単位の滞在秒数を再計算する正本にはならない。Desktop 側のアクティブウィンドウ情報は、あくまで OS 全体の文脈把握や fallback 用途に留める。
- そのため、同じブラウザ行動について `閲覧クエスト` 用と `行動ログ` 用で別々の滞在時間カウンタを持たず、一次ソースは Chrome 拡張の計測結果を再利用する。
- Chrome 拡張が利用できない環境では、ブラウザ前面化の粗い文脈だけが `RawEvent` に残る場合があるが、詳細なドメイン別滞在時間や閲覧クエスト判定の正本には使わない。

### 5-5. リリィデスクトップ5分記録での再利用

- `lily_desktop` は `Activity Capture Service` が保持する privacy 適用済みの直近 `RawEvent` snapshot を、5分ごとの状況記録や desktop 雑談種生成の入力として再利用してよい。
- このとき生成する `デスクトップ行動要約` は、直近5分の行動ログをもとにした軽量な自然文サマリであり、本仕様の `ActivitySession` とは別概念とする。
- `デスクトップ行動要約` は server-side action-log entity として保存を必須にせず、状況ログや雑談の補助情報として扱う。
- 生成AIは local `gemma4` 系モデルを使ってよく、応答形式は JSON ではなく日本語テキストのみを正本とする。
- AI 失敗時は `appName` / `domain` / `projectName` / idle 状態などから rule-based fallback を生成し、状況記録全体を止めない。

## 6. ログの階層

### 6-1. RawEvent

- 最下層の出来事ログ。
- サーバー保存時は TTL 対象とし、既定では `30日` で自動削除する。
- 用途は直近ログの再セッション化、要約再生成、トラブルシュート補助に限定し、長期閲覧の正本にはしない。
- 例:
  - `2026-04-16T10:03:00+09:00 Chrome で YouTube を開いた`
  - `2026-04-16T10:18:00+09:00 VS Code が前面化した`

### 6-2. ActivitySession

- 近いイベントを束ねた作業単位。
- 例:
  - `10:00-10:40 Chrome拡張の調査`
  - `14:10-15:00 VS Codeでクエスト一覧修正`
- 基本はルールベースで束ね、必要に応じて AI がタイトルや補助要約を付ける。

### 6-3. DailyActivityLog

- 1 日分の流れをまとめた読み返し単位。
- 含むもの:
  - `その日のまとめ`
  - `クエストクリア状況まとめ`
  - `健康状況まとめ`
  - いずれもリリィがユーザーのその日をそっと見守って書いた観察日記風の地の文
  - 補助情報としての `mainThemes` / `reviewQuestions`
  - 手動メモとの関連付け

### 6-4. WeeklyActivityReview

- 週単位での傾向分析と振り返り。
- 含むもの:
  - リリィがユーザーの一週間の流れを見守って綴る観察日記・観察メモ風の地の文
  - カテゴリ別時間配分
  - よく出てきたテーマ
  - 増えた活動、減った活動
  - 中断したままの作業
  - 来週のための軽い問いかけ

### 6-5. SituationLog（30分まとめ）

- `lily_desktop` が直近 30 分の 5 分記録から作る補助要約。
- `ActivitySession` とは別概念・別 API で保存するが、`/records/activity/today` と `/records/activity/day/:dateKey` の `session` 表示では読み返し用に併記してよい。
- 同日分を `timestamp` の新しい順に表示する。
- `event` 表示では出さない。
- 長期的に読み返せるよう、server 保存時に TTL は付けない。

## 7. セッション化と分類

### 7-1. セッション化ルール

- 同一アプリまたは同一ドメイン上の連続したイベントは、5 分未満の中断であれば同一セッションにまとめてよい。
- 次の場合はセッションを分割する。
  - アプリが切り替わった
  - ドメインが大きく切り替わった
  - 作業対象のプロジェクトまたはファイルが変わった
  - 5 分以上の idle が入った
- v0.1 ではユーザーによるセッション手動分割やタイトル修正は提供しない。補正は非表示と手動メモ追加を中心とする。
- AI は既に束ねられた候補セッションの `タイトル付け`, `要約`, `分類補助` を担当し、未加工の生イベント取得は担当しない。

### 7-2. 一次カテゴリ

- 行動ログの一次カテゴリは既存の閲覧カテゴリと揃え、以下の 8 種を正本とする。
  - `学習`
  - `仕事`
  - `健康`
  - `生活`
  - `創作`
  - `対人`
  - `娯楽`
  - `その他`
- 判定基準は `spec/閲覧カテゴリ分類ガイド.md` を正本とし、行動ログ側も可能な限り同じ意味で扱う。

### 7-3. 二次ラベル

- 一次カテゴリとは別に、セッション単位でより具体的な `activityKind` を持てる。
- 例:
  - `開発`
  - `調査`
  - `学習`
  - `会議`
  - `執筆`
  - `雑務`
  - `休憩`
  - `移動`

## 8. AI の責務

### 8-1. AI が担当すること

- セッションに人が読めるタイトルを付ける
- 一次カテゴリと二次ラベルを補助判定する
- カレンダー予定や手動メモから文脈を補完する
- 過去ログの意味検索用インデックスを作る
- 振り返り用の問いを生成する

### 8-2. 通常の行動ログ処理モデル

- 行動ログ基盤の通常 AI 処理は、ローカル実行の `gemma4` 系モデルを使う。
- v0.1 の既定値は Ollama 上の `gemma4:e4b` とし、`http://127.0.0.1:11434` を既定の接続先とする。
- `activity_processing.enabled` の既定値は `true`、`activity_processing.provider` の既定値は `ollama`、`activity_processing.base_url` の既定値は `http://127.0.0.1:11434`、`activity_processing.model` の既定値は `gemma4:e4b`、`activity_processing.max_completion_tokens` の既定値は `400` とする。
- 以下は原則として local `gemma4` が担当する。
  - セッションタイトル生成
  - 一次カテゴリ / 二次ラベル判定
  - 手動メモや予定による文脈補完
  - 検索補助用のクエリ解釈や結果要約
- `ActionLogOrganizeRequested` を organizer 起動契約とし、desktop 起動時 1 回と `activity_capture.sync_interval_seconds` ごとの timer tick を起点に best-effort 実行する。
- organizer は server 上の `RawEvent` を読まず、`lily_desktop/logs/action_logs/raw_events/YYYY-MM-DD.jsonl` にある現在 device の `today + yesterday` 分の local spool を再読して `ActivitySession` を再構成する。
- organizer が `activity_processing.provider=openai` を使う場合、uncached candidate は `startedAt` の新しい順に優先し、最大 8 session ごとに Structured Outputs へ送る。1 run あたりの OpenAI enrichment はおおむね 60 秒で打ち切り、残りは rule-based fallback で埋めて `PUT /action-log/sessions` まで必ず進める。
- organizer stats には少なくとも `reused_count`, `ai_count`, `fallback_count`, `batch_count`, `budget_exhausted` を残し、OpenAI budget に達した run は専用ログで識別できるようにする。
- この層では、できる限りローカル処理を優先し、クラウド送信を前提にしない。

### 8-3. 日次・週次まとめモデル

- `DailyActivityLog.summary`, `DailyActivityLog.questSummary`, `DailyActivityLog.healthSummary` の自然文要約は `OpenAI gpt-5.4` を使う。
- `WeeklyActivityReview` の自然文要約は `OpenAI gpt-5.4` を使う。
- `DailyActivityLog` は `summary`, `questSummary`, `healthSummary` をそれぞれ別々の `gpt-5.4` リクエストで生成する。PWA 前日 day 画面の自動生成と desktop 起動時 backfill は同じ 3 分割契約を使う。
- PWA / desktop の日次 3 リクエストと desktop の `WeeklyActivityReview` backfill では `max_output_tokens=1600` を明示し、出力上限不足で `incomplete(reason=max_output_tokens)` になりにくいようにする。
- 日次まとめの入力は JST 同日の構造化データを正本とし、`summary` は `ActivitySession`、`questSummary` は `QuestCompletion` と関連 `Quest` 情報、`healthSummary` は `health-data` を主入力にしつつ `fitbit-data` と `nutrition-data` も同日データとして入力に含める。
- 週次まとめは、raw event の全文をそのまま送るのではなく、事前に整形済みの `ActivitySession` と必要最小限のメタ情報を入力に使う。
- スクリーンショット画像、メール本文、チャット本文、絶対パスなどの機微情報は日次・週次まとめ生成の入力に含めない。
- 文体は `リリィがユーザーを見守って書いた観察日記` を基本とする。チャットのように直接話しかける口調ではなく、落ち着いた地の文でその日 / その週の流れをやわらかく記述する。
- 文体は責めず、決めつけず、少し親しみはあるが踏み込みすぎない観察者トーンとする。提案や励ましは補足としてよく、本文は観察記述を主とする。
- 日次まとめでは、`その日の流れ`, `クエストの達成状況`, `健康状況` を優先してまとめる。
- 週次まとめでは、`良かった流れ`, `主要テーマ`, `次に見ると良い観点` を優先してまとめる。

### 8-4. AI が担当しないこと

- 生イベントの取得
- OS 入力の監視
- 自動でクエスト完了や評価を確定すること
- プライバシールールを無視した全文送信

### 8-5. フォールバック

- local `gemma4` が無効、未設定、失敗時でも `RawEvent` と `ActivitySession` の基本保存は止めない。
- local `gemma4` が使えない場合は、ルールベースの簡易タイトル、カテゴリ推定、テンプレート補完でフォールバックする。
- `DailyActivityLog` はテンプレート文へフォールバックしない。失敗した section は未生成のまま保持し、desktop 次回起動時または PWA で該当日を開いた次回に missing section だけ再試行する。
- `WeeklyActivityReview` の `gpt-5.4` 呼び出しができない場合は、テンプレート文へフォールバックする。
- v0.1 では、日次・週次まとめのフォールバック先として local `gemma4` へ自動切り替えは行わない。

## 9. 画面仕様

### 9-1. 画面一覧

- タイムライン
- デイリーログ
- カレンダー
- 検索
- 週次行動レビュー一覧
- 週次行動レビュー詳細
- 設定

### 9-2. タイムライン

- 日付単位で時系列ログを表示する。
- `event` 単位と `session` 単位を切り替えられる。
- `/records/activity/today` と `/records/activity/day/:dateKey` のタイムラインは、新しい時刻が上に来る降順表示とする。`session` 表示は `startedAt`、`event` 表示は `occurredAt` を基準にする。
- `/records/activity/today` と `/records/activity/day/:dateKey` のタイムラインは、client が日別全件を先読みせず、server の paged read で `50件` ずつ読む。初期表示は active view (`session` または `event`) の最新 `50件` のみを取得し、残りは `さらに50件表示` で次ページを取得する。
- `today/day` では `session` と `event` の page state を分け、最初に開いていない側のタイムラインは初回表示では取得しない。view 切り替え時に必要になった時点で 1 ページ目を読む。
- `today/day` のタイムライン footer では総件数を出さず、`nextCursor` がある時だけ `さらに50件表示` を出す。
- `today/day` の `Target` カードでは、モバイルでも `対象日` と `session / event` 切り替えを同じ行に収める。
- `today/day` の `session` 表示では、`DailyActivityLog` の 3 本まとめの下に、同日の `SituationLog` を `timestamp` 降順で表示し、その下に session timeline を置く。
- `today/day` の `event` 表示では `SituationLog` を出さず、`DailyActivityLog` の下に event timeline だけを置く。
- セッションのタイトル、時間帯、カテゴリ、主なアプリ / ドメインを表示する。
- `today/day` の `session` 表示では、60 秒以内の近接・3 分以内の短時間バーストに含まれる session を、domain/app 単位の UI 専用 compact block として束ねてよい。compact は表示専用であり、保存データ、search、export、週次レビュー、`event` 表示は変えない。
- `today/day` のセッションカードでは直接 `非表示 / 再表示` ボタンを出さず、右上は時刻アイコン中心の軽い表示とする。hidden session の確認や復元は検索画面で行う。

### 9-3. デイリーログ

- `today/day` では `DailyActivityLog` を timeline より先に表示する。
- `today/day/calendar/search/review` 系の行動ログ共通ナビには `今日`, `昨日`, `カレンダー`, `検索`, `週次レビュー` をこの順で置く。`昨日` は JST 前日の `/records/activity/day/:dateKey` へのショートカットとし、`今日` / `昨日` の切り替えでは現在の `view=session|event` を維持する。
- `DailyActivityLog` は少なくとも `その日のまとめ`, `クエストクリア状況まとめ`, `健康状況まとめ` の 3 本を持ち、上からこの順に表示する。
- 3 本ともリリィ観察日記風の地の文とし、`questSummary` と `healthSummary` も `gpt-5.4` を使って生成する。
- `mainThemes` / `reviewQuestions` はデータとして保持してよいが、v0.1 の `today/day` 画面では必須表示にしない。
- `today/day` では `DailyActivityLog` と `ActivitySession` を中心に表示し、途中状態専用の別エンティティは持たない。
- v0.1 ではタイトルや要約の手動修正は持たず、必要に応じて手動メモ追加や非表示化で補助する。

### 9-4. カレンダー

- カレンダーは 1 画面につき 1 か月分を表示する。
- 既定表示は JST 基準の当月とする。
- `前月` / `次月` ボタンで月移動できる。
- `年月ピッカー` で任意の年月へ直接移動できる。
- 日付セルには `DailyActivityLog.summary` だけを表示する。`mainThemes` などの補助 badge は v0.1 では出さなくてよい。
- 日付セルを押すと `/records/activity/day/:dateKey` へ遷移する。

### 9-5. 検索

- キーワード、カテゴリ、アプリ、ドメイン、期間、タグで過去ログを絞り込める。
- 意味検索は AI が有効な場合のみ使える。

### 9-6. 週次行動レビュー

- 主入口は年ごとの一覧とする。
- 一覧画面は 1 画面につき 1 年分の `WeeklyActivityReview` を表示する。
- 既定表示は JST 基準の当年、または直近の利用可能年とする。
- `前年` / `次年` ボタンで年移動できる。
- `年ピッカー` で任意の年へ直接移動できる。
- 一覧では各週の短い要約、主要テーマ、カテゴリ比率の要点を表示する。
- 週カードを押すと `/records/activity/review/week?weekKey=YYYY-Www` の詳細画面へ遷移する。
- 詳細画面では 1 週間分の本文、カテゴリ比率、継続テーマ、よく使ったアプリ / ドメインを表示する。
- 既存の `週次ふりかえり` とは別画面とする。

### 9-7. 設定

- 収集 ON / OFF
- デバイスごとの記録状態
- 除外アプリ
- 除外ドメイン
- URL 全文保存 / ドメインのみ保存
- AI 利用範囲
- ログのエクスポート
- 期間削除、一括削除

### 9-8. 正式 route と導線

- 行動ログ系の正式 route は以下とする。
  - `/records/activity/today?view=session|event`
  - `/records/activity/day/:dateKey?view=session|event`
  - `/records/activity/calendar?month=YYYY-MM`
  - `/records/activity/search`
  - `/records/activity/review/year?year=YYYY`
  - `/records/activity/review/week?weekKey=YYYY-Www`
- Web アプリ内の通常導線でも、desktop の右クリック導線でも同じ route を使う。
- BottomNav の `記録` は `/records` を開き、その後に `records` 配下の最後に閲覧していた route を復元する。
- 初回表示または復元不能時の既定 route は `/records/quests?range=today` とする。
- `行動ログ` タブの既定 route は `/records/activity/today` とする。
- `記録` 画面上部の `成長記録 / 行動ログ` タブと、行動ログ内の `今日 / カレンダー / 検索 / 週次レビュー` pill ナビは、選択中状態で文字色とアイコン色のコントラストを十分に確保する。少なくとも暗色背景には白文字を使い、視認性を落とさない。
- `review/year` の各週カードの導線ラベルは `詳細` とし、モバイルでは本文と衝突しない縦積みレイアウトを許容する。
- `/records/activity/calendar` を `month` query なしで開いた場合は、JST 基準の当月を表示する。
- `/records/activity/review/year` を `year` query なしで開いた場合は、JST 基準の当年または直近の利用可能年を表示する。
- `/records/activity/review/year` は `year` query により表示年を切り替える。`前年` / `次年` / `年ピッカー` はこの query を更新して実現する。
- `/records/activity/review/week` は `weekKey` query により表示週を切り替える詳細 route とし、年一覧からの遷移先として使う。
- desktop から開く URL は安定した path を優先し、日付や検索語のような一時的 state は必要時のみ query に載せる。

## 11. プライバシー仕様

- 行動ログ収集はユーザーの明示的な有効化を前提とする。
- デバイス単位で `収集中`, `一時停止`, `無効` を切り替えられる。
- シークレット / プライベートブラウジングは記録しない。
- メール、チャット、SNS の DM、銀行、決済、認証、個人情報画面は既存のデスクトップ状況ルールに従って除外する。
- URL は `全文保存` と `ドメインのみ保存` を選択できる。
- ファイル情報は既定で `projectName` と `fileName` までに留め、絶対パスの保存は既定で行わない。
- AI へ送る前に匿名化または要約だけに落とす設定を持てる。
- リリィデスクトップの5分記録と雑談生成では、デスクトップスクリーンショットを定期取得しない。手動取得が残る場合も長期保存はしない。
- ログ削除、期間削除、エクスポートをサポートする。
- 非表示にしたセッションは通常の画面と Lily の会話コンテキストから除外する。
- privacy rule の優先順位は `window_title > domain > app > storage_mode` とし、同一 type 内では `updatedAt` が新しいものを優先する。

### 11-1. 保存期間と TTL

- `RawEvent` はサーバー側で TTL 対象とし、既定保持期間は `30日` とする。
- `RawEvent` の TTL 起点は `occurredAt` を基準とし、保存時に `expiresAt` を付与してよい。
- TTL により `RawEvent` が削除されても、既に生成済みの `ActivitySession`, `DailyActivityLog`, `WeeklyActivityReview`, `ManualNote`, `SituationLog` は保持し続ける。
- `ActivitySession`, `DailyActivityLog`, `WeeklyActivityReview`, `ManualNote`, `SituationLog`, `PrivacyRule`, `Device` は既定で TTL を設けず、ユーザー削除または明示的な期間削除まで保持する。
- UI と Lily が長期的に参照する正本は `ActivitySession` 以上と `SituationLog` とし、`RawEvent` は最近の詳細確認用データとして扱う。
- v0.1 では `RawEvent` の保持期間は固定値 `30日` とし、ユーザーごとの保持日数カスタマイズは必須要件にしない。
- DynamoDB など TTL 削除が遅延しうる基盤では、期限到達後すぐに物理削除されない場合があってもよい。

### 11-2. Action Log / Related APIs

- v0.1 の action-log API は `/activity-logs` とは分離して `/action-log/...` 名前空間で提供する。
- `RawEvent` の read/write は `POST /action-log/raw-events` と `GET /action-log/raw-events?from=YYYY-MM-DD&to=YYYY-MM-DD` を正本とする。
- `ActivitySession` の read/write は `PUT /action-log/sessions` と `GET /action-log/sessions?from=YYYY-MM-DD&to=YYYY-MM-DD` を正本とする。`PUT` は payload に含まれる `dateKey` 群に対する full-replace とする。
- `ActivitySession.searchKeywords` は検索 UI と Lily 参照で使う検索補助語彙として保持する。
- `DailyActivityLog` は `GET /action-log/daily/{dateKey}` / `PUT /action-log/daily/{dateKey}` を正本とし、一覧取得は `GET /action-log/daily?from=YYYY-MM-DD&to=YYYY-MM-DD` を使う。
- `WeeklyActivityReview` は `GET /action-log/weekly/{weekKey}` / `PUT /action-log/weekly/{weekKey}` を正本とし、年一覧は `GET /action-log/weekly?year=YYYY` を使う。
- `Device` は `GET /action-log/devices` と `PUT /action-log/devices/{id}`、`PrivacyRule` は `GET /action-log/privacy-rules` と `PUT /action-log/privacy-rules` を正本とする。
- `SituationLog` は `GET /situation-logs?from=YYYY-MM-DD&to=YYYY-MM-DD` と `POST /situation-logs` を正本とする。行動ログ画面では `session` 表示の補助ログとして読む。
- `GET /action-log/daily/{dateKey}` と `GET /action-log/weekly/{weekKey}` は未生成時に `null`、range/list API は未存在時に `[]` を返す。

## 12. 日次・週次レビュー仕様

### 12-1. 日次ログ生成

- `/records/activity/today` では当日の `DailyActivityLog` を自動生成しない。保存済みの `DailyActivityLog` がある場合だけ表示し、未生成時は空表示のままとする。
- `/records/activity/day/:dateKey` では `dateKey` が JST の前日と一致する場合に限り、missing section を含む `DailyActivityLog` を自動生成してよい。
- `/records/activity/day/:dateKey` で前日以外の過去日は `DailyActivityLog` を自動生成しない。
- PWA 前日 day 画面では `summary`, `questSummary`, `healthSummary` を並列で開始し、成功した section から既存 `DailyActivityLog` にマージして保存する。生成中の section は `生成中…` を表示し、失敗 section は `まだ生成されていません。次回また生成します。` として扱う。
- `lily_desktop` 起動時は raw event sync と organizer 完了後に summary backfill を 1 回だけ実行し、JST の前日 `DailyActivityLog` に missing section があればその section だけ補完する。
- desktop のデバッグメニューには `前日の DailyActivityLog を再生成` を置き、`summary`, `questSummary`, `healthSummary` の 3 本を強制再実行できるようにする。失敗 section は既存値を保持する。
- 同じ `dateKey` の complete な `DailyActivityLog` は通常 backfill では再生成しない。missing section がある場合だけ次回起動や該当日表示時に再試行し、手動更新では 3 section すべて再生成してよい。
- 生成モデルは `OpenAI gpt-5.4` 固定とする。
- 生成処理は v0.1 では PWA クライアント側と `lily_desktop` 起動時 backfill で行い、既存の OpenAI API 設定を使う。
- 生成内容は `summary`, `questSummary`, `healthSummary` の 3 本を正本とし、必要に応じて `mainThemes` / `reviewQuestions` を補助出力してよい。
- `summary` は当日の `ActivitySession` を主入力にし、`questSummary` は JST 同日の `QuestCompletion` と関連 `Quest` 情報、`healthSummary` は JST 同日の `health-data` を主入力にしつつ `fitbit-data` と `nutrition-data` も合わせて入力する。
- 文体は「今日は午前中、...」のようなリリィ観察日記風の地の文を正本とし、クエストと健康のまとめも同じ観察者トーンでそろえる。

### 12-2. 週次行動レビュー生成

- Web クライアントで `WeeklyActivityReview` を自動生成してよいのは JST 月曜日だけとし、対象は `previousWeekKey` に一致する前週レビューだけに限る。
- `/records/activity/review/year?year=YYYY` は保存済み `WeeklyActivityReview` を一覧表示する画面を正本とし、JST 月曜日に `year` が `previousWeekKey` の ISO week-year と一致する場合だけ、未生成の前週レビュー 1 件を補完してよい。
- `/records/activity/review/week?weekKey=YYYY-Www` は週詳細画面として扱い、JST 月曜日かつ `weekKey === previousWeekKey` の場合に限り、未生成の前週レビュー 1 件を補完してよい。
- `/records/activity/review/week` で任意の未生成週をその場で生成する挙動は v0.1 では採用しない。
- `/records/activity/review/year?year=YYYY` は前週補完条件に当てはまらない限り、保存済み `WeeklyActivityReview` だけを一覧表示し、未生成週の一括生成は行わない。
- `lily_desktop` 起動時は曜日に関係なく raw event sync と organizer 完了後に summary backfill を 1 回だけ実行し、`previousWeekKey` の `WeeklyActivityReview` が未生成なら補完する。
- 既存の `週次ふりかえり` とは混ぜない。
- 生成モデルは `OpenAI gpt-5.4` 固定とする。
- 生成処理は v0.1 では PWA クライアント側と `lily_desktop` 起動時 backfill で行い、既存の OpenAI API 設定を使う。
- 文体は「今週は、...」のようにリリィがユーザーの一週間を見守って綴る観察日記風の地の文とし、責めず、時間配分と流れの把握を優先する。

## 13. 既存機能との関係

- `quests` / `completions` / `skills` は従来どおり成長記録の正本とする。
- Chrome 拡張の閲覧データは、`閲覧クエスト` と `行動ログ` の両方で利用してよい。
- `週次ふりかえり` は引き続き成長記録中心とし、本仕様の `WeeklyActivityReview` を別系統で持つ。
- リリィの Tool Search で参照する `activity_logs` は、本仕様の `ActivitySession` と `DailyActivityLog` を中心に返す。
- `activity_logs` の実装では、Web 側 Lily と `lily_desktop` 側 Lily の両方が action-log API を正本として参照する。
- `activity_logs` では `hidden=true` の `ActivitySession` を会話コンテキストから除外する。
- リリィの会話では raw event をそのまま読み上げず、要約済みのセッションや日次ログを優先して使う。
- 本仕様の `RawEvent` / `ActivitySession` は、リリィデスクトップの `デスクトップ行動要約` の入力や fallback 情報として再利用してよい。
- `today/day` の `session` 表示では、action-log 系データに加えて `situation_logs` の 30 分まとめを補助表示してよい。

## 14. 非機能要件

- 記録取得はユーザー操作を妨げない軽さを優先する。
- 収集失敗や AI 失敗があっても、他の主要機能を止めない。
- 同期失敗時はローカル記録を保持し、次回同期に回す。
- 生ログよりセッション、セッションより日次ログを優先して閲覧させる。
- 仕様上の優先順位は `安心して使えること > 思い出せること > きれいに要約されること` とする。
## 15. Phase 3 実装メモ

- `Activity Capture Service` は v0.1 で `RawEvent` を `lily_desktop/logs/action_logs/raw_events/YYYY-MM-DD.jsonl` に JSONL で durable spool し、`lily_desktop/logs/action_logs/sync_state.json` に日別ファイルごとの `ackedLines` を保存する。
- `activity_capture.enabled` の既定値は `true`、`activity_capture.initial_state` の既定値は `active`、`activity_capture.poll_interval_seconds` の既定値は `2`、`activity_capture.sync_interval_seconds` の既定値は `30` とする。
- Local HTTP Bridge は `browser_page_changed` / `heartbeat` を正式受理し、desktop 側の `RawEvent` 系列へ正規化してよい。
- desktop 起点の `file context` は v0.1 では空許容とし、`projectName` / `fileName` は未設定のまま保存してよい。

## 16. Phase 4 実装メモ

- `RawEvent` のサーバー同期は `ActionLogSyncRequested` を起点に行い、起動時 1 回と 30 秒ごとの定期同期で `/action-log/raw-events` へ best-effort 送信する。
- 同期ジョブは `single_flight_coalesce` を正本とし、同時多発した同期要求は 1 本にまとめる。
- `RawEvent` の server-side TTL は `expiresAt` を Unix epoch 秒へ変換した `ttl` を DynamoDB へ保存することで実現する。`ActivitySession` 以上の構造化データには TTL を付けない。
- action-log API は `/action-log/raw-events`, `/action-log/sessions`, `/action-log/daily`, `/action-log/weekly`, `/action-log/devices`, `/action-log/privacy-rules` を正式 route とする。
- `RawEvent` は event id を sort key に含めた upsert を前提とし、desktop 側の再送で同一 event が来てもサーバー側で重複行を増やさない。

## 17. Phase 5 実装メモ

- `lily_desktop` は `ActionLogOrganizer` を持ち、`RawEvent -> ActivitySession` の生成を正本として担当する。
- organizer は `today + yesterday` の local spool を再読し、5 分以上の gap、`idle_started` / `idle_ended`、主要 `domain` 変更、`projectName` / `fileName` 変更、browser 系以外の `appName` 変更をもとに rule-based で candidate session を分割する。
- `chrome.exe` / `msedge.exe` / `firefox.exe` / `brave.exe` / `opera.exe` / `vivaldi.exe` / `arc.exe` と `chrome_extension` source は同一 browser bucket として扱い、同一 domain・短時間内なら 1 つの candidate session に束ねる。
- `idle_started` / `idle_ended` は separator event として扱い、session 本体には含めない。
- `ActivitySession.id` は `sorted(rawEventIds)` を素材にした deterministic hash とし、title や category に依存させない。
- organizer は `PUT /action-log/sessions` で sessions を同期しつつ、同じ id の既存 session が `hidden=true` ならその値を維持する。
- organizer の AI enrichment は title / summary / activityKinds / searchKeywords を日本語で生成する。GitHub や YouTube などの固有名は含めてよいが、韓国語（Hangul）が含まれる enrichment は再利用せず、当該 candidate は rule-based fallback の日本語文面で保存する。
- organizer の user-facing 文言では `heartbeat`, `browser_page_changed`, `active_window_changed`, `raw event` などの内部 telemetry 名を出さない。既存 session の再利用結果や新規 AI enrichment にこれらが含まれる場合も fallback に切り替える。
- local `gemma4` が失敗した場合でも、rule-based fallback で session title / primaryCategory / activityKinds / summary / searchKeywords を埋めて保存を継続する。

## 18. Phase 9 実装メモ（2026-04-18）

### 18-1. `/settings` の行動ログ設定
- `/settings` には「行動ログ」セクションを追加し、`Device.captureState`、除外アプリ、除外ドメイン、URL 保存粒度、AI 利用範囲を編集できるようにする。
- settings UI は local draft + 明示 Save とし、device は `PUT /action-log/devices/{id}`、privacy rule は `PUT /action-log/privacy-rules` の full-replace で保存する。
- URL 保存粒度は `PrivacyRule { type: "storage_mode", value: "default_url_storage" }` に正規化して保存する。
- AI 利用範囲は `PrivacyRule { type: "storage_mode", value: "default_ai_handling" }` に正規化して保存する。`normal` を選んだ場合はこの rule を保存しない。

### 18-2. desktop runtime 設定同期
- Phase 9 以降、runtime の `captureState` と `privacy_rules` の正本は server とする。`config.yaml` は bootstrap default としてだけ使う。
- `lily_desktop` は起動時と 30 秒 sync tick ごとに `GET /action-log/devices` と `GET /action-log/privacy-rules` を取り込み、自分の device を見つけたら runtime に反映する。
- server 上に自分の device が存在しない場合は、local config の初期 state を使って `PUT /action-log/devices/{id}` で登録する。
- `captureState = active` は通常収集、`paused` は service を生かしたまま収集停止、`disabled` は capture service を停止する。
- `ActivityCaptureService` は `set_privacy_rules()` を持ち、server から取得した rule を runtime に差し替える。

### 18-3. 非表示・検索・エクスポート・削除
- `ActivitySession.hidden` は targeted update で変更できる。API は `PUT /action-log/sessions/{id}/hidden`、body は `{ dateKey, hidden }` とする。
- `today / day / search` では session を非表示にできる。`day / search` では hidden session を再表示できる。
- 既定の一覧と Lily の `activity_logs` では `hidden=true` の session を除外する。search では `includeHidden` を明示したときだけ含める。
- search は `keyword / from / to / categories / apps / domains / includeHidden` を持つ厳密検索とし、対象は `ActivitySession` とする。
- action-log export は client-side JSON bundle とし、`rawEvents`, `sessions`, `dailyLogs`, `weeklyReviews`, `situationLogs`, `meta` を含める。
- action-log delete は search 画面から実行し、`to <= JST の昨日` の範囲だけを許可する。当日分は UI から削除できない。
- delete 対象は `RawEvent`, `ActivitySession`, `DailyActivityLog`, `WeeklyActivityReview`, `SituationLog` とし、quests / completions / messages / settings など既存データは対象外とする。

### 18-4. purge queue
- `DELETE /action-log/range?from=YYYY-MM-DD&to=YYYY-MM-DD` は server 上の action-log entity と同期間の `SituationLog` を削除したうえで purge request を 1 件作る。
- purge request API は `GET /action-log/deletion-requests` と `POST /action-log/deletion-requests/{id}/ack` を正本とする。
- desktop は未処理 purge request を受けたら `raw_events/YYYY-MM-DD.jsonl` と `sync_state.json` を purge し、完了後に ack する。
- これにより、削除済み期間が raw sync や organizer から再送・再生成されない状態を保つ。
