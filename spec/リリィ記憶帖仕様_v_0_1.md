# リリィ記憶帖仕様 v0.1

更新日: 2026-04-25

## 0. 位置づけ

本仕様は、リリィがユーザーの会話・行動・クエスト・健康状態から長期的な文脈を自動的に覚え、会話や提案に活用する `リリィ記憶帖` を定義する。

現行の `lily_desktop` にある `memory_*.txt` を使った `思い出` 雑談種は「リリィ自身の思い出」を扱う仕組みであり、本仕様の対象外とする。

本仕様の `記憶` は、ユーザーについてリリィが育てる長期記憶である。

## 1. 目的

- リリィがユーザーの興味、目標、習慣、作業スタイル、配慮事項、進行中テーマを継続的に理解できるようにする。
- 会話、雑談、クエスト提案、リマインド、ふりかえりで、毎回ゼロから文脈を読み直さずに自然な応答を行えるようにする。
- ユーザーが明示的に登録しなくても、日々のログからリリィの理解が育つ状態を作る。
- ただし、ユーザーが後から見直し、修正、削除、アーカイブ、拒否できる状態を保つ。

## 2. 体験方針

- 記憶登録は `明示承認制` ではなく `自動登録制` とする。
- リリィは、会話や行動ログをもとに自動で記憶を作成・更新してよい。
- UI は「登録する場所」ではなく「あとから眺めて調整する場所」とする。
- ユーザーが毎回確認しなくても使えることを優先する。
- 違和感があれば、ユーザーは会話または画面操作で後から修正できる。
- リリィは記憶を使ってよいが、断定しすぎず、特に弱い記憶は `最近は` `たぶん` `〜の傾向がありそう` のように扱う。

## 3. 非目的

- 性格診断や人格評価を行うこと。
- ユーザーを採点・格付けすること。
- 削除できないプロフィールを作ること。
- 他ユーザーとの共有、ランキング、公開プロフィール化。
- 医療的判断や専門的診断を記憶として確定すること。
- 生ログ全文を長期記憶として保存すること。

## 4. 入力ソース

記憶抽出は、以下の保存済みデータを入力にしてよい。

| sourceType | 入力内容 | 主な用途 |
| --- | --- | --- |
| `chat` | Web リリィチャット | 好み、目標、嫌なこと、興味 |
| `desktop_chat` | リリィデスクトップでの会話 | 好み、作業文脈、配慮事項 |
| `activity_log` | `ActivitySession` | 習慣、作業スタイル、進行中テーマ |
| `daily_log` | `DailyActivityLog` | その日の傾向、健康・クエスト文脈 |
| `weekly_review` | `WeeklyActivityReview` | 繰り返し出るテーマ、長期傾向 |
| `quest` | Quest | 目標、継続したい行動 |
| `completion` | Completion | 習慣、達成傾向、得意な粒度 |
| `skill` | Skill | 育っている分野、興味領域 |
| `fitbit` | Fitbit 日次サマリー | 睡眠・活動傾向 |
| `nutrition` | 食事・栄養データ | 食生活の傾向 |
| `healthplanet` | 体重・体脂肪率 | 体重計測習慣、健康文脈 |
| `browsing` | 閲覧カテゴリ・ドメイン時間 | 興味、学習テーマ、注意閲覧傾向 |
| `situation_log` | 30 分まとめ | 作業環境・生活文脈 |

`RawEvent` は短期保持の生ログであり、v0.1 の記憶抽出では直接の長期証拠として保存しない。必要な場合も、抽出後の `evidenceSummary` には要約済みの情報のみを残す。

## 5. 記憶タイプ

```ts
type LilyMemoryType =
  | 'interest'
  | 'goal'
  | 'routine'
  | 'project'
  | 'preference'
  | 'boundary'
  | 'work_style'
  | 'health_pattern'
  | 'relationship'
```

| type | 意味 | 例 |
| --- | --- | --- |
| `interest` | 興味分野 | 健康、心理学、React、仕事術に関心がある |
| `goal` | 目標 | 睡眠を整えたい、React を継続的に学びたい |
| `routine` | 習慣・傾向 | 夜に創作作業が進みやすい |
| `project` | 進行中テーマ | 自分育成アプリの改善を進めている |
| `preference` | 好み | 朝は軽い提案のほうがよい |
| `boundary` | 避けたいこと・配慮事項 | 強い通知は避けたい |
| `work_style` | 作業スタイル | 調査してから実装する流れが多い |
| `health_pattern` | 健康・生活リズム傾向 | 睡眠が短い日は行動量が落ちやすい |
| `relationship` | 人間関係・交流傾向 | イベント参加や交流に関心がある |

## 6. 記憶ステータス

```ts
type LilyMemoryStatus =
  | 'tentative'
  | 'active'
  | 'archived'
  | 'rejected'
```

| status | 意味 | AI 利用 |
| --- | --- | --- |
| `tentative` | 自動抽出直後の仮の記憶 | 弱めに利用してよい |
| `active` | 通常利用する記憶 | 通常文脈として利用してよい |
| `archived` | 今は使わない記憶 | 利用しない |
| `rejected` | 間違いとして拒否された記憶 | 利用しない |

`tentative` は候補ではなく、利用可能な仮記憶とする。ただし応答では断定を避ける。

## 7. データモデル

```ts
interface LilyMemory {
  id: string
  type: LilyMemoryType
  status: LilyMemoryStatus
  title: string
  body: string
  evidenceSummary: string
  tags: string[]
  sourceTypes: LilyMemorySourceType[]
  sourceRefs: LilyMemorySourceRef[]
  confidence: number
  strength: number
  sensitivity: 'normal' | 'sensitive'
  useInChat: boolean
  useInSuggestions: boolean
  createdAt: string // JST RFC3339
  updatedAt: string // JST RFC3339
  lastSeenAt: string // JST RFC3339
  lastUsedAt?: string // JST RFC3339
  expiresAt?: string // JST RFC3339
}

type LilyMemorySourceType =
  | 'chat'
  | 'desktop_chat'
  | 'activity_log'
  | 'daily_log'
  | 'weekly_review'
  | 'quest'
  | 'completion'
  | 'skill'
  | 'fitbit'
  | 'nutrition'
  | 'healthplanet'
  | 'browsing'
  | 'situation_log'

interface LilyMemorySourceRef {
  sourceType: LilyMemorySourceType
  sourceId?: string
  occurredAt?: string // JST RFC3339
  dateKey?: string // YYYY-MM-DD (JST)
}
```

### フィールド方針

- `title` は一覧表示用の短い名前。
- `body` はリリィが会話で使う自然文の記憶内容。
- `evidenceSummary` は `なぜ覚えたか` の表示に使う短い根拠要約。
- `sourceRefs` は元データへの参照であり、生ログ本文を保存しない。
- `confidence` は抽出確度。0.0 〜 1.0。
- `strength` は繰り返し確認された強さ。0.0 〜 1.0。
- `sensitivity='sensitive'` の記憶は、会話で使う場合も直接的な言い回しを避ける。
- `useInChat=false` の記憶はチャット・雑談の prompt に入れない。
- `useInSuggestions=false` の記憶はクエスト提案・リマインドに使わない。

## 8. 自動抽出タイミング

### 8-1. チャット後抽出

Web / Desktop の会話後、ユーザー発話とリリィ応答から記憶候補を抽出する。

- 好き / 苦手 / 続けたい / 避けたい / 目標 / 興味 / 作業テーマを拾う。
- 1 ターンだけの軽い話題は原則 `tentative` にする。
- `覚えておいて` のような明示発話がある場合は `active` にしてよい。
- `これは嫌` `これは避けたい` のような強い境界表現は `boundary` として `active` にしてよい。

### 8-2. 日次ログ後抽出

`DailyActivityLog` 生成後、その日の安定した行動傾向を抽出する。

- `ActivitySession` 由来の作業テーマ。
- `questSummary` 由来の継続クエスト。
- `healthSummary` 由来の健康・睡眠・食事傾向。
- 1 日だけの傾向は原則 `tentative` とする。

### 8-3. 週次レビュー後抽出

`WeeklyActivityReview` 生成後、複数日にまたがって出現したテーマを記憶として強める。

- 既存記憶と近い場合は新規作成ではなく `strength` と `lastSeenAt` を更新する。
- 繰り返し出ている `routine`, `project`, `work_style`, `interest` は `active` に昇格してよい。

### 8-4. クエスト / 完了記録からの抽出

クエスト作成、完了、繰り返し達成から記憶を更新する。

- 同じカテゴリやスキルの完了が続く場合は `routine` または `goal` として扱う。
- 長期間継続しているクエストは `active` にしやすい。
- 完了しなくなったクエスト由来の記憶は、一定期間後に `archived` 候補にしてよい。

### 8-5. 健康・食事データからの抽出

Fitbit、Health Planet、Nutrition の日次データから `health_pattern` を抽出する。

- 医療的判断はしない。
- `睡眠が短い日が続いている` `歩数が少ない日が続いている` のような観察に留める。
- 会話では断定や診断ではなく、行動候補の調整に使う。

## 9. 抽出 AI 仕様

記憶抽出には、低コストの構造化出力モデルを使う。

- v0.1 の候補モデルは `gpt-5-nano` とする。
- 失敗時は記憶抽出だけをスキップし、会話・ログ保存・クエスト処理は止めない。
- 出力は JSON Structured Outputs とする。
- 抽出結果は最大件数を制限し、1 回の処理で大量の記憶を作らない。

### 9-1. 抽出出力

```ts
interface LilyMemoryExtractionResult {
  memories: Array<{
    type: LilyMemoryType
    title: string
    body: string
    evidenceSummary: string
    tags: string[]
    confidence: number
    sensitivity: 'normal' | 'sensitive'
    mergeHint?: string
  }>
}
```

### 9-2. 抽出禁止

以下は記憶にしない。

- パスワード、認証コード、秘密鍵、トークン。
- メール本文やチャット本文の詳細な引用。
- 銀行、決済、医療、認証画面の具体的な内容。
- 一時的な感情を恒久的な性格として断定する内容。
- 他者の個人情報を必要以上に含む内容。

## 10. マージ・昇格ルール

### 10-1. 重複判定

新しい抽出結果は、既存の `active` / `tentative` 記憶と比較し、近い内容なら新規作成せず更新する。

比較材料:

- `type`
- `tags`
- `title`
- `body`
- `mergeHint`
- source の種類

### 10-2. 昇格

- `confidence >= 0.75` は `active` にしてよい。
- `confidence >= 0.4 && confidence < 0.75` は `tentative` にする。
- 7 日以内に 3 回以上、近い source で確認された記憶は `active` に昇格する。
- `boundary` は 1 回でも強い表現なら `active` にしてよい。
- `health_pattern` は原則として複数日確認後に `active` にする。

### 10-3. 弱体化・アーカイブ

- `routine` / `project` が 60 日以上 `lastSeenAt` 更新されない場合は `archived` 候補にしてよい。
- `interest` / `preference` は長期に残してよいが、1 年以上使われない場合は見直し候補にする。
- `boundary` は自動アーカイブしない。

## 11. 会話・提案での利用

### 11-1. 取得方針

リリィの応答生成時、関連しそうな記憶だけを選んで prompt に入れる。

- 毎回すべての記憶を入れない。
- `active` と `tentative` のみ対象にする。
- `archived` / `rejected` は使わない。
- `useInChat=false` の記憶はチャット・雑談に使わない。
- `useInSuggestions=false` の記憶はクエスト提案・リマインドに使わない。
- 1 回の prompt に入れる記憶は最大 8 件を目安にする。

### 11-2. 優先順位

優先度:

1. `boundary`
2. `preference`
3. 会話内容に直接関係する `project` / `goal`
4. 直近更新された `routine` / `work_style`
5. `interest`
6. `health_pattern`

### 11-3. 表現ルール

- `tentative` の記憶は断定せず、弱い表現にする。
- `sensitive` の記憶は、ユーザーから直接話題にした場合を除き、露骨に言わない。
- 記憶を使った場合でも、ユーザーの現在の発話を最優先する。
- 記憶に反する発話があった場合は、記憶の更新または拒否を検討する。

## 12. UI 仕様

### 12-1. ルート

正式 route は `/lily/memories` とする。

導線:

- リリィチャットのサイドドロワー。
- 設定画面の `リリィ記憶帖` セクション。
- Desktop の `Web を開く` メニューには v0.1 では必須にしない。

### 12-2. 一覧

表示グループ:

- 最近覚えたこと
- よく使われる記憶
- 興味
- 目標
- 習慣
- 作業テーマ
- 好み
- 配慮事項
- 健康・生活リズム

各カードに表示する項目:

- `title`
- `body`
- `type`
- `status`
- `confidence` / `strength`
- `updatedAt`
- `なぜ覚えたか` (`evidenceSummary`)
- source 種別の短い表示

### 12-3. 操作

各記憶で以下を操作できる。

- 編集
- 削除
- アーカイブ
- `これは違う` (`rejected`)
- `これは大事` (`active` かつ `strength` を上げる)
- `AI会話で使う / 使わない`
- `提案で使う / 使わない`

削除は物理削除とし、以後の prompt に入れない。

## 13. 会話からの修正

ユーザーは画面を開かなくても、会話で記憶を修正できる。

例:

- `それ覚えないで`
- `今のは忘れて`
- `それ違う`
- `その記憶は消して`
- `これは覚えておいて`
- `これは大事`

処理方針:

- `それ違う` は直近で使用した記憶を `rejected` にする。
- `忘れて` / `消して` は対象が明確なら削除する。不明確なら確認する。
- `覚えておいて` は該当内容を `active` で作成または更新する。
- 会話からの修正は `lastUsedAt` と直近 prompt 記憶を参照して解決する。

## 14. API 仕様

### 14-1. エンドポイント

| メソッド | パス | 説明 |
| --- | --- | --- |
| GET | `/lily-memories` | 記憶一覧取得 |
| POST | `/lily-memories` | 記憶作成 |
| PUT | `/lily-memories/{id}` | 記憶更新 |
| DELETE | `/lily-memories/{id}` | 記憶削除 |
| POST | `/lily-memories/extract` | 指定入力から記憶抽出 |
| POST | `/lily-memories/retrieve` | 会話・提案向け関連記憶取得 |

### 14-2. 認証

既存の Cognito JWT 認証を使用する。

### 14-3. DynamoDB スキーマ

```
PK: user#${userId}
SK: LILY_MEMORY#${memoryId}
```

検索補助として、必要に応じて以下の GSI を持ってよい。

```
GSI1PK: LILY_MEMORY#${userId}#${status}
GSI1SK: ${updatedAt}
```

## 15. 設定

`Settings` に以下を追加してよい。

```ts
interface LilyMemorySettings {
  enabled: boolean
  autoExtractFromChat: boolean
  autoExtractFromActivityLog: boolean
  autoExtractFromHealth: boolean
  useInChat: boolean
  useInSuggestions: boolean
}
```

既定値:

- `enabled: true`
- `autoExtractFromChat: true`
- `autoExtractFromActivityLog: true`
- `autoExtractFromHealth: true`
- `useInChat: true`
- `useInSuggestions: true`

## 16. エラー時の扱い

- 記憶抽出に失敗しても、元の会話・クエスト・ログ保存は成功扱いを維持する。
- 記憶 API が失敗した場合、リリィ応答は記憶なしで継続する。
- 記憶取得が空の場合は、従来の会話コンテキストだけで応答する。
- 記憶抽出の失敗は JST の runtime / activity log に記録してよいが、ユーザーに毎回通知しない。

## 17. v0.1 実装範囲

v0.1 で含める:

- `LilyMemory` の保存 API。
- Web / Desktop のリリィ会話からの自動抽出。
- `DailyActivityLog` / `WeeklyActivityReview` からの自動抽出。
- `completion` 反復からの `routine` / `goal` 更新。
- `active` / `tentative` の自動判定。
- リリィチャットの system prompt への関連記憶注入。
- `/lily/memories` の一覧・編集・削除・アーカイブ・拒否 UI。
- `それ違う` / `忘れて` / `覚えておいて` の会話操作。

v0.1 で含めない:

- ベクトル検索基盤。
- 他ユーザーとの共有。
- 記憶の公開プロフィール化。
- RawEvent 全文からの直接長期記憶作成。
- 医療判断や診断としての健康記憶。

## 18. テスト観点

- チャットから `interest` / `goal` / `boundary` が抽出される。
- `boundary` は 1 回でも `active` になる。
- 弱い推測は `tentative` になる。
- 週次レビューで同じテーマが繰り返されると `active` に昇格する。
- 既存記憶と近い抽出結果は新規作成ではなく更新される。
- `rejected` / `archived` は prompt に入らない。
- `useInChat=false` はチャット prompt に入らない。
- `それ違う` で直近使用記憶が `rejected` になる。
- `忘れて` で対象記憶が削除される。
- 記憶抽出 API が失敗しても会話送信は失敗しない。
- JST の `createdAt` / `updatedAt` / `lastSeenAt` が保存される。
