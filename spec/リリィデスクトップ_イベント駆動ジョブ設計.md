# リリィデスクトップ イベント駆動 + ジョブ管理 設計

## 1. 目的

リリィデスクトップ内部の背景処理を、UI シグナルと分離した内部イベント駆動 + ジョブ管理で整理する。

今回の対象は `lily_desktop` 内部のみとし、Chrome 拡張、Web、Local HTTP Bridge の外部契約は変更しない。

## 2. 二層構成

### 2-1. UI signal bus

- `core/event_bus.py` を UI 専用の signal bus として維持する
- 吹き出し表示、ポーズ変更、音声 ON/OFF、手動ボタン入力など、PySide6 の UI に直接関わる通知を担当する

### 2-2. internal event/job layer

- `core/domain_events.py` に typed event を定義する
- `core/background_event_runtime.py` に handler 登録を集約する
- `core/job_manager.py` で背景処理の実行ポリシーを統一する

この層では UI 更新を行わず、背景処理の起動順序、重複抑止、再実行制御のみを扱う。

## 3. イベント一覧

### 3-1. Phase 1 で導入するイベント

| イベント | 用途 | 主な発火元 |
| --- | --- | --- |
| `AppStarted` | 起動後の背景処理オーケストレーション開始 | `async_init()` |
| `HealthPlanetSyncRequested` | Health Planet 同期要求 | 起動時、定期タイマー |
| `FitbitSyncRequested` | Fitbit 同期要求 | 起動時 |
| `ChatAutoTalkDue` | 自動会話要求 | 自動会話タイマー、手動トリガー |

### 3-2. Phase 4 で追加するイベント

| イベント | 用途 | 主な発火元 |
| --- | --- | --- |
| `ChatFollowUpRequested` | ユーザー発話後の follow-up 会話要求 | チャット応答後 |
| `CaptureSnapshotRequested` | 5 分ごとの状況取得要求 | カメラタイマー |
| `CaptureSummaryDue` | 30 分サマリ送信要求 | サマリタイマー |

## 4. ジョブポリシー

| ポリシー | 意味 | 適用対象 |
| --- | --- | --- |
| `single_flight_drop` | 実行中なら新規要求を捨てる | 自動会話、follow-up |
| `single_flight_coalesce` | 実行中なら 1 回だけ再実行予約する | Health Planet 同期、Fitbit 同期 |
| `latest_wins` | 古い未完了ジョブをキャンセルして新しい要求に置き換える | 5 分キャプチャ |
| `serial` | 要求を順番に直列実行する | 30 分サマリ |

### 4-1. 定期自動雑談の保留ルール

- `ChatAutoTalkDue` のうち `source="auto_conversation.timer"` かつ通常会話中のものは、job manager へ即投入しない
- `main.py` が最新 1 件だけを保留し、FIFO キューにはしない
- 保留の期限は `occurred_at + 5分`（JST）とし、期限内に通常会話と follow-up が終わったら `chat.auto_talk` を 1 回だけ投入する
- 期限切れの保留は破棄し、UI 通知は出さずログだけを残す
- 定期自動雑談を実行する直前に、直近 90 秒以内の `chrome_audible_tabs` snapshot に `chat.auto_talk_skip_audible_domains` 一致の domain があれば、その回をスキップする
- 手動発火や `forced_source` 付きの自動会話はこの保留ルールに入れず、そのまま実行する

## 5. 責務分割

### 5-1. `core/domain_events.py`

- `DomainEvent` 基底型を定義する
- 共通項目は `event_id`, `occurred_at`, `source`, `correlation_id` とする
- `occurred_at` は JST の timezone-aware `datetime` に正規化する

### 5-2. `core/job_manager.py`

- `submit(job_key, policy, runner)` を公開入口とする
- ジョブの重複抑止、coalesce、cancel、直列化を担当する
- UI 表示やドメイン判断は行わない

### 5-3. `core/background_event_runtime.py`

- internal event と job policy の対応付けを行う
- `ChatAutoTalkDue` は `main.py` の会話調停へ委譲し、定期自動雑談の保留判定もそこで行う
- `main.py` から背景処理の直呼びを減らし、handler 登録をこのファイルへ寄せる

### 5-4. `main.py`

- event hub と job manager の生成、および各コンポーネントへの注入を行う
- 通常会話の busy 区間を管理し、定期自動雑談の保留・破棄・drain を担当する
- 既存の UI signal bus との接着は維持する

### 5-5. `ai/auto_conversation.py`

- 会話本体の実行と interrupt の責務を持つ
- タイマー到達時は、会話本体を直接起動せず internal event を publish する

## 6. 実装方針

- まず基盤を導入し、既存挙動を壊さずに起動時同期をイベント化する
- 次に 15 分おしゃべりをイベント化する
- 最後に follow-up と状況取得をイベント化し、handler 登録を専用モジュールへ集約する

## 7. テスト方針

- 追加仕様は TDD で進める
- Phase ごとに failing test を先に追加する
- 関連既存テストと新規テストを実行し、green を確認したうえで TODO を更新する

## 8. 補足（2026-04-05 更新）
- `main.py` の `async_init()` は evented 版を正本とし、起動時の背景処理 publish はこの 1 系統に統一する。
- `_capture_and_record` / `_fetch_and_show_desktop_context` / `_debug_capture_and_record` は coordinated 実装への薄い委譲に保ち、未到達の旧実装は残さない。
