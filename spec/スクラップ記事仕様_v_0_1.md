# スクラップ記事仕様 v0.1

更新日: 2026-05-02

## 0. 位置づけ

本仕様は、Web 上の記事 URL を `あとで読む / あとで見返す` ために保存する `スクラップ記事` 機能を定義する。

スクラップ記事は、行動ログやクエストとは独立した記録データであり、Web アプリ（PWA）から閲覧・追加・整理できる。

Android では、PWA の Web Share Target API を利用し、ブラウザや他アプリの共有メニューから記事を保存できるようにする。iOS では v0.1 時点で共有メニューからの PWA 直接保存は対象外とし、Web アプリ内の手動追加を正規導線とする。

## 1. 目的

- Android の共有メニューから、記事 URL とタイトルを素早く保存できるようにする。
- Web アプリ内からも URL を手動追加できるようにする。
- 保存した記事を `記録` 配下で一覧し、元記事を開けるようにする。
- 記事本文取得や要約に踏み込まず、まずは軽量な URL スクラップ機能として成立させる。

## 2. 非目的

- v0.1 では、記事本文の取得・全文保存・本文リーダー表示は行わない。
- v0.1 では、記事本文の AI 要約・自動タグ付け・カテゴリ分類は行わない。
- v0.1 では、iOS の共有シートから PWA を直接共有先として表示することは保証しない。
- v0.1 では、ネイティブ iOS アプリ、iOS Share Extension、Capacitor 等のラッパーアプリは作らない。
- v0.1 では、複数人共有、公開コレクション、SNS 的な共有機能は扱わない。

## 3. 配置

スクラップ記事機能は、Web アプリの `記録` 配下に配置する。

| 画面 | パス | 役割 |
| --- | --- | --- |
| スクラップ一覧 | `/records/scraps` | 保存済み記事の一覧、読了切り替え、外部ページ起動 |
| 手動追加 | `/records/scraps/new` | URL、タイトル、メモを入力して保存 |
| 共有受け取り | `/?shareTarget=article` | Android 共有メニューから起動される内部入口 |

`共有受け取り` は通常ユーザーが直接操作する画面ではない。共有データを検証・一時退避し、保存後に `/records/scraps` へ遷移する。

現行 Web アプリは HashRouter を使うため、Web Share Target の `action` は HashRouter の `#/...` ではなく、通常 URL のクエリで受けることを基本とする。

例:

```json
{
  "share_target": {
    "action": "./?shareTarget=article",
    "method": "GET",
    "params": {
      "title": "title",
      "text": "text",
      "url": "url"
    }
  }
}
```

## 4. Android 共有からの追加

### 4-1. 利用条件

- ユーザーが Android の Chrome 等から PWA をインストール済みであること。
- PWA manifest に `share_target` が定義されていること。
- 共有元アプリが URL、タイトル、テキストのいずれかを共有していること。

### 4-2. 基本フロー

1. ユーザーが Android のブラウザや記事アプリで共有メニューを開く。
2. 共有先として `自分育成アプリ` を選ぶ。
3. PWA が `/?shareTarget=article` で起動する。
4. アプリが `title` / `text` / `url` を読み取る。
5. URL とタイトルを正規化する。
6. ログイン済みであれば DB に保存する。
7. 未ログインであれば共有データを一時保存し、ログイン後に保存する。
8. 保存成功、重複、失敗の結果を表示し、最終的に `/records/scraps` へ遷移する。

### 4-3. 受信データの解釈

Android の共有では、共有元アプリやブラウザによって `url` が空になり、URL が `text` または `title` に入る場合がある。そのため、次の優先順で URL を決定する。

1. `url` に有効な `http:` / `https:` URL があれば採用する。
2. `text` に URL が含まれていれば最初の URL を採用する。
3. `title` に URL が含まれていれば最初の URL を採用する。
4. URL が決定できない場合は保存せず、手動追加画面へ誘導する。

タイトルは次の優先順で決定する。

1. `title` があり、URL だけではない場合は採用する。
2. `text` から URL を除いた文字列があれば採用する。
3. URL のドメインを仮タイトルとして使う。

共有元から届いた生の `title` / `text` / `url` は、必要に応じて `sourceText` に保存してよい。ただし長文本文の保存は v0.1 の目的ではないため、過度に長い文字列は上限長で切り詰める。

## 5. Web アプリ内の手動追加

`/records/scraps` に追加ボタンを置き、`/records/scraps/new` へ遷移する。

手動追加画面は次の入力を持つ。

| 項目 | 必須 | 内容 |
| --- | --- | --- |
| URL | 必須 | `http:` / `https:` の記事 URL |
| タイトル | 任意 | 空の場合はドメインを仮タイトルにする |
| メモ | 任意 | ユーザーの短いメモ |

URL 入力時に、v0.1 では外部ページへアクセスしてタイトルを自動取得することを必須としない。CORS や取得失敗の影響を避けるため、タイトルは共有データまたはユーザー入力を正とする。

## 6. スクラップ一覧

`/records/scraps` で保存済み記事を一覧表示する。

表示項目:

- タイトル
- ドメイン
- 追加日
- ステータス
- メモの先頭部分

操作:

- 記事をタップすると元記事 URL を開く。
- `すべて` / `未読` / `読了` / `アーカイブ` で表示を絞り込む。
- 未読 / 読了を切り替える。
- アーカイブする。
- 削除する。
- 手動追加画面へ移動する。

元記事 URL は PWA 内本文ビューではなく、ブラウザまたは同一タブの外部ページとして開く。実装時は `target="_blank"` を使う場合、`rel="noopener noreferrer"` を付ける。

## 7. データモデル

論理モデルは次の通り。

```ts
type ScrapArticleStatus = 'unread' | 'read' | 'archived'

type ScrapArticleAddedFrom = 'android-share' | 'manual'

interface ScrapArticle {
  id: string
  url: string
  canonicalUrl: string
  title: string
  domain: string
  sourceText?: string
  memo?: string
  status: ScrapArticleStatus
  addedFrom: ScrapArticleAddedFrom
  createdAt: string // JST RFC3339
  updatedAt: string // JST RFC3339
  readAt?: string // JST RFC3339
  archivedAt?: string // JST RFC3339
}
```

### 7-1. フィールド方針

- `id` はアプリ内で一意な ID。
- `url` はユーザーが保存した元 URL。
- `canonicalUrl` は重複判定に使う正規化 URL。
- `title` は一覧表示用のタイトル。
- `domain` は一覧表示、検索、将来の分類に使う。
- `sourceText` は Android 共有から届いた補助テキスト。長文本文保存には使わない。
- `memo` はユーザーが任意で入力する短いメモ。
- `status='unread'` を初期値とする。
- `createdAt` / `updatedAt` / `readAt` / `archivedAt` は JST 基準で扱う。

### 7-2. URL 正規化

重複判定用の `canonicalUrl` は、少なくとも次の正規化を行う。

- URL 前後の空白を除去する。
- `http:` / `https:` 以外は保存対象外にする。
- ホスト名を小文字化する。
- URL フラグメントは除去する。
- 末尾の不要な `/` は、意味が変わらない範囲で揃える。

クエリパラメータは記事識別に必要な場合があるため、v0.1 では原則保持する。将来、`utm_source` などのトラッキングパラメータ除去を追加してよい。

## 8. 保存先と同期

スクラップ記事はログインユーザーごとのデータとして DB に保存する。

現行アプリの方針に合わせ、クライアントはローカル状態を先に更新し、クラウド同期は API で行う。オフラインや API 失敗時の再送キューは v0.1 では必須としないが、主要操作が失敗した場合はユーザーにわかる表示を出す。

想定 API:

| 操作 | メソッド | パス | 内容 |
| --- | --- | --- | --- |
| 一覧取得 | `GET` | `/scraps` | ログインユーザーのスクラップ一覧を取得 |
| 作成 | `POST` | `/scraps` | スクラップ記事を作成 |
| 更新 | `PUT` | `/scraps/{id}` | タイトル、メモ、ステータスを更新 |
| 削除 | `DELETE` | `/scraps/{id}` | スクラップ記事を削除 |

DynamoDB 上では `PK=user#<Cognito sub>`, `SK=SCRAP#<id>` として保存する。

## 9. 重複仕様

同一ユーザー内で `canonicalUrl` が同じ記事は重複とみなす。

重複時の挙動:

- 新規レコードは作成しない。
- 既存記事を表示対象にする。
- 既存記事の `createdAt` は変更しない。
- `updatedAt` は、メモやステータスなど明示的な更新がある場合のみ変更する。
- Android 共有から重複保存された場合は、`保存済みの記事です` と表示し、一覧の該当記事へ誘導する。

## 10. 未ログイン時の扱い

Android 共有から起動した時点で未ログインの場合、共有データを失わないことを優先する。

- 受信した共有データを `sessionStorage` に一時保存する。
- ログイン画面へ誘導する。
- ログイン成功後、一時保存した共有データを読み取り、保存処理を再開する。
- 保存成功またはユーザーキャンセル後、一時保存データを削除する。

一時保存データには URL、タイトル、共有テキストのみを含め、認証情報や不要な個人情報を含めない。

## 11. iOS の扱い

v0.1 では、iOS 共有シートから PWA を直接共有先として表示することは仕様対象外とする。

iOS での記事保存導線は次の通り。

1. Safari 等で記事 URL をコピーする。
2. Web アプリを開く。
3. `/records/scraps/new` で URL を貼り付ける。
4. 必要に応じてタイトルやメモを入力して保存する。

将来 iOS の共有シートから直接保存したい場合は、ネイティブ iOS アプリまたはラッパーアプリに iOS Share Extension を実装し、共有された URL を API 経由で保存する。その場合は別仕様で定義する。

## 12. エラー表示

| ケース | 表示方針 |
| --- | --- |
| URL が見つからない | `URLを読み取れませんでした。URLを貼り付けて追加してください。` |
| URL 形式が不正 | `保存できるURLではありません。` |
| 未ログイン | ログイン画面へ誘導し、ログイン後に保存を再開 |
| DB 保存失敗 | `保存できませんでした。通信状態を確認してもう一度お試しください。` |
| 重複 | `保存済みの記事です。` |

## 13. v0.1 完了条件

- Android 共有メニューから PWA を選び、URL とタイトルを保存できる。
- Android 共有で `url` が空でも、`text` から URL を抽出して保存できる。
- 未ログイン時に共有データが失われず、ログイン後に保存できる。
- Web アプリ内の手動追加画面から URL を保存できる。
- `/records/scraps` で保存済み記事を一覧表示できる。
- `すべて` / `未読` / `読了` / `アーカイブ` で絞り込める。
- 一覧から元記事 URL を開ける。
- 未読 / 読了を切り替えられる。
- 同一 URL の重複保存を防げる。
- iOS では手動追加が正規導線であることが仕様上明記されている。
