# リリィデスクトップ for Windows仕様

## 1. コンセプト
* リリィの立ち絵を、Windowsデスクトップ右下に常駐するデスクトップマスコットとして表示する。
* 相方はリリィの左側に立ち、二人で掛け合いしながらデスクトップ上で会話する。
* ユーザーは二人の掛け合いを眺めることができ、必要に応じて会話に参加できる。
* 性格はWebアプリ側のリリィと一致させる。

## 2. 表示・UI仕様
* 常時大きなアプリwindowは表示せず、透明背景でキャラクターと吹き出しのみをデスクトップ上に表示する。
* リリィは画面右下に立ち、相方はその左側に立つ。
* リリィのセリフはバルーンで表示する。(lily_desktop/sys/sys_images/message_windows.png)
* リリィと相方の発話は、それぞれのキャラクターに紐づいた吹き出しで表示する。
* メッセージ表示の際は `【リリィ】` のように名前を `【】` で囲んで1行目に表示し、2行目以降に `「やっほ～」` のようにセリフを `「」` で囲んで表示する。
* ユーザー入力はテキスト入力か音声入力で受け付ける。
* 入力UIは必要時のみ最小限の形で表示する。
* リリィと相方のセリフ内容に応じて、その場に合った立ち絵へ切り替える。

## 3. 会話仕様
* ユーザーはテキスト入力または音声入力で、リリィと会話できる。
* Webアプリで利用できるtool searchは、デスクトップマスコット版のリリィでも利用できる。
* 相方がいて、リリィとよく掛け合いを行う。
* ときたまリリィ、または相方から話しかけ、二人の会話が自然に始まることがある。
* 二人の会話は10ターン程度を想定する。
* ユーザーは、二人の会話を見ている途中から会話に参加できる。
* 15分タイマー起点の定期自動雑談は、通常会話中やその follow-up 中に割り込ませない。
* 定期自動雑談のタイミングが通常会話中に来た場合は、最新1件だけ保留し、通常会話と follow-up の完了後に1回だけ実行する。
* 保留した定期自動雑談が 5 分を超えて古くなった場合は、その回は破棄する。
* 定期自動雑談を実行する直前に、Chrome の可聴タブ情報を確認する。
* `chat.auto_talk_skip_audible_domains` に一致する可聴タブが直近 90 秒以内に報告されている場合、その回の定期自動雑談はスキップする。
* 上記の可聴タブ判定は 15 分タイマー起点の定期自動雑談にのみ適用し、手動雑談・本雑談・クエスト雑談・follow-up には適用しない。

## 4. 音声入力
* 随時受け付けているが、webrtcなどのvadのゲートがある
* 音声認識はgoogleのSTTを使う
* ドキュメント : https://docs.cloud.google.com/speech-to-text/docs?hl=ja
* `voice.pause_during_tts` の既定値は `true` とし、既定では TTS 再生中にマイク入力を一時停止する。
* `voice.pause_during_tts: false` を設定した場合は、TTS 再生中もマイクを止めず、ユーザー音声で会話に割り込みできる。
* TTS再生後に音声入力を再開する際は、保存済みマイク名から毎回デバイスを再解決する。
* 保存済みマイクで再開できない場合は、その回に限ってデフォルト入力デバイスへフォールバックする。
* デフォルト入力でも再開できない場合は、そのセッションでは音声入力の自動再開を停止し、明示的なマイク再選択または音声入力の再起動でのみ復帰する。

## 5. デスクトップ状況システム
* デスクトップ上での自然な雑談のために、その時点のデスクトップ状況を会話の種として利用する。
* リリィは必要に応じて、その時のデスクトップのスクリーンショットを取得して状況を把握できる。
* 本アプリは個人利用前提とし、ユーザーが明示的に許可している場合に限り、スクリーンショット内容を会話生成に利用してよい。
* スクリーンショットは常時連続取得ではなく、雑談候補を作るタイミングで必要時のみ取得する。
* カメラ状況取得とデスクトップ状況取得は共有コーディネータで管理し、同時実行しない。
* まずアクティブウィンドウのアプリ名・ウィンドウタイトル・ブラウザの場合はドメインを確認し、解析対象にしてよい画面か判定する。
* メール、チャット、SNSのDM、パスワードマネージャ、銀行、決済、認証コード表示画面、個人情報や機密情報を扱う画面は解析対象から除外する。
* ブラウザはアプリ単位ではなく、表示中のドメイン単位で解析可否を判定する。
* 解析対象外と判定された場合はスクリーンショット解析を行わず、そのタイミングでは別の雑談種を使う。
* 状況取得中に同種の取得要求が来た場合は待機せずスキップし、追加のスクリーンショット解析やカメラ解析は行わない。
* スクリーンショット解析AIは provider を切り替え可能にする。`desktop.analysis_provider` が `openai` のときは OpenAI Chat Completions を使い、`ollama` のときは Ollama `/api/chat` を使う。
* この仕様更新時点の既定値は `desktop.analysis_provider: ollama` / `desktop.analysis_model: gemma4:e4b` / `desktop.analysis_base_url: http://127.0.0.1:11434` とする。
* スクリーンショット解析で Ollama を使う場合は、構造化された最終回答を優先するため `think=false` で呼び出し、応答上限は `num_predict=500` とする。
* 解析対象の画面では、スクリーンショットをAIで読んで「今はコーディング中」「調べもの中」「動画視聴中」「作業が止まっている」などの状況要約を作る。
* スクリーンショットから得た内容は、そのまま全文を読み上げるためではなく、雑談用の状況要約と話題タグに変換するために使う。
* 雑談に使う際は、画面上の文章やコードをそのまま引用するのではなく、画面の状況や行動傾向を自然な会話に変換して扱う。
* スクリーンショット画像は長期保存せず、状況要約と話題タグのみを保持する。
* 状況要約の例は「コーディングしている」「ドキュメントを読んでいる」「調べものをしている」「動画で休憩している」「しばらく操作が止まっている」などとする。

## 6. 雑談システム
* リリィと相方の雑談は、複数の情報源から集めた「雑談の種」をもとに生成する。
* 雑談の主な種は以下の10系統とする。
* デスクトップ状況: その時のスクリーンショットやアクティブウィンドウの状況から作られる、現在進行形の話題。
* カメラ状況: カメラ映像から取得した周囲の状況（天気・人物・動物・室内外など）をもとにした話題。
* 豆知識（注目記事）: Wikimedia から取得する「今日は何の日」「注目記事」「軽い雑学」などの話題。
* 豆知識（興味分野）: ユーザーの興味ある分野をもとに Wikimedia から取得したWikipedia記事の話題。広いテーマは内部で複数の関連サブトピックへ展開し、集めた候補からランダムに選んで幅広い知識を持ってくる。
* エンタメ: Annict から取得するアニメ作品、放送中作品、エピソード、視聴候補などの話題。
* 健康豆知識: Fitbitの活動・睡眠・心拍データおよび食事・栄養素データをもとに生成する健康に関する話題。歩数・睡眠時間・安静時心拍のサマリや、不足・過剰栄養素をもとに豆知識として話す。
* 本: 楽天Books Book Search API から固定6ジャンル（自己啓発・習慣・心理学・仕事術・健康・小説）を毎回1つランダムに選び、そのジャンルの売れ筋上位20件を取得して説明文 (`itemCaption`) をもとに生成する話題。レビュー本文は使わない。必要に応じて `Origin` ヘッダーを付けて取得する。
* 思い出: `talk_seeds.memory_directory` で指定したフォルダ内の `memory_*.txt` をランダムに読み込み、リリィ自身の思い出として自然に語る話題。本文の丸読みはせず、内容を思い出話としてやわらかく言い換えて使う。話し始めは「〜の話なんだけど」のように、何の思い出かが最初に伝わる切り出しを入れる。`talk_seeds.memory_directory` の既定値は `D:\codes\mixi2-api\generated_text` とする。空文字のときは本カテゴリを無効化する。相対パスを指定した場合は `lily_desktop/config.yaml` 基準で解決する。
* クエスト今日状況: 今日のクエスト完了状況と現在の進行中クエストをもとにした振り返り話題。
* クエスト週次状況: 直近7日のクエスト完了状況と現在の進行中クエストをもとにした振り返り話題。
* Wikimedia の仕様URL:
  * Feed API Overview: https://api.wikimedia.org/wiki/Feed_API
  * Feed API Reference: https://api.wikimedia.org/wiki/Feed_API/Reference
  * Featured content: https://api.wikimedia.org/wiki/Feed_API/Reference/Featured_content
* Annict の仕様URL:
  * API Overview: https://docs.annict.com/docs/ja/api/v1/overview
  * Authentication: https://docs.annict.com/docs/ja/api/v1/authentication
  * Works API: https://docs.annict.com/docs/ja/api/v1/works
  * Episodes API: https://docs.annict.com/docs/ja/api/v1/episodes
* 楽天Books の仕様URL:
  * Books Book Search API: https://webservice.rakuten.co.jp/documentation/books-book-search
* 雑談生成時は、10系統のカテゴリを均等確率（各約10%）で先に抽選し、当たったカテゴリの種だけを取得する。
* 抽選されたカテゴリのデータが取得できなかった場合は、そのカテゴリを除外して残りのカテゴリで再抽選する。全カテゴリがクールダウン中だった場合に限り、クールダウン中の種をフォールバックとして再利用してよい。
* 雑談の種は「要約」「話題タグ」「鮮度」「出典」「リリィ側の切り口」「相方側の切り口」を持つカードとして扱う。
* リリィは雑談の種を見て話を振り、相方はそれに対してツッコミ、補足、脱線、同意などを返す。
* ユーザーが会話に入りやすいよう、必要に応じて最後に軽い問いかけを入れる。
* キャラクターの右クリックメニューのデバッグから `楽天Books雑談を開始` を選ぶと、本カテゴリ限定で雑談を即時発火できる。楽天キー未設定や候補不足のときは本カテゴリをスキップする。
* キャラクターの右クリックメニューのデバッグから `思い出雑談を開始` を選ぶと、思い出カテゴリ限定で雑談を即時発火できる。`talk_seeds.memory_directory` が未設定、フォルダ不存在、候補不足のときは思い出カテゴリをスキップする。
* キャラクターの右クリックメニューのデバッグから `今日のクエスト雑談を開始` を選ぶと、クエスト今日状況カテゴリ限定で雑談を即時発火できる。クエスト状況を取得できないときはその回をスキップする。
* キャラクターの右クリックメニューのデバッグから `週次クエスト雑談を開始` を選ぶと、クエスト週次状況カテゴリ限定で雑談を即時発火できる。クエスト状況を取得できないときはその回をスキップする。
* デスクトップ状況を使った雑談では、画面の文面をそのまま読み上げるのではなく、「集中してるね」「今日は調べものモードだね」のように柔らかく要約して話す。
* カメラ状況を使った雑談では、「カメラに映ってる」「画像で見ると」のようなメタな言い方を避け、二人がその場の様子を見ているような自然な言い回しで話す。
* Wikimedia の話題は、重いニュースではなく日常会話に混ぜやすい軽い豆知識を優先する。
* Wikimedia の興味分野トピックは、`科学` のような広い語をそのまま単純検索するのではなく、関連するサブトピック群へ展開して候補を広げ、同じ記事タイトルは重複候補にしない。
* Annict の話題は、アニメの新着、季節作品、話題作、見どころなど、相方との掛け合いにしやすい内容を優先する。
* 政治、事件、事故、対立を煽る話題、ユーザーの不安を強める話題は雑談の種として優先しない。
* 同じ雑談の種を短時間で繰り返さないようにし、使った話題は一定期間クールダウンさせる。
* 雑談の種が複数ある場合は、時間帯、直前の会話、デスクトップ状況との相性を見て最も自然なものを選ぶ。
* デスクトップ状況と外部話題を組み合わせて、「今コードを書いてるみたいだけど、そういえば今日はこういう話題もあるよ」のような自然な橋渡しをしてもよい。
* Wikimedia / Annict / 楽天Books / 健康データの取得に失敗した場合でも、デスクトップ状況またはカメラ状況だけで雑談を継続できるようにする。
* 健康豆知識の種は1日1回クールダウン（同日中に同じ健康データから重複して話さない）する。

## 7. データ管理
* リリィとユーザーの会話はDBに保存される。
* リリィと相方の会話もDBに保存される。
* DBに保存された会話データを参照し、過去のやり取りを踏まえた会話ができる。
* webアプリの会話も参照できる
* webアプリからもデスクトップでの会話を参照できる
* 例: 「そういえば昨日こういう話をしていて〜」のように、過去会話を踏まえた発話を行える。
* 画像系のファイルは全てローカルのみで管理する。
* デスクトップ状況システムで取得したスクリーンショット画像は長期保存せず、必要なら状況要約・話題タグ・取得時刻のみを保存する。
* 雑談で使った種の出典、要約、使用履歴は再利用や重複防止のために保存してよい。

## 8. ポーズ生成システム

### 8-1. AIレスポンス形式
* AIからのセリフレスポンスはJSON形式とする: `{"text": "セリフ", "pose_category": "joy"}`
* `pose_category` にはポーズカテゴリ名を指定する。
* 通常会話・自動雑談の両方で同一形式を使う。

### 8-2. ポーズカテゴリ
* ポーズは以下の共通カテゴリ（8種）とリリィ専用カテゴリ（5種）に分類する。
* 共通カテゴリはリリィ・相方の両方で使用する。
* リリィ専用カテゴリは成長パートナーとしての役割に特化した表情。

**共通カテゴリ（8種）**

| カテゴリ名 | 意味 |
|-----------|------|
| default | 通常 |
| joy | 喜び・嬉しい |
| anger | 怒り・抗議 |
| sad | 哀しみ |
| fun | 楽しい・笑い |
| shy | 照れ |
| worried | 悩み・心配 |
| surprised | 驚き |

**リリィ専用カテゴリ（5種）**

| カテゴリ名 | 意味 |
|-----------|------|
| proud | 得意・自信 |
| caring | 気遣い・心配 |
| serious | 真剣・集中 |
| sleepy | 眠い・疲れ |
| playful | いたずら・ウィンク |

### 8-3. リリィのポーズ管理
* リリィは各カテゴリにつき5種類のバリエーションを持つ（合計65枚）。
* 立ち絵は `lily_desktop/sys/lily_images/` に保存する。
* 対応関係ファイル（`lily_pose_map.json`）でカテゴリ・ファイル名の対応を管理する。
* セリフのpose_categoryに一致するカテゴリから、ランダムまたは順番に1枚を選択する。

### 8-4. リリィのポーズ生成（不足分の自動生成）
* カテゴリに対応する立ち絵がない場合、または5種に満たない場合は、都度 `gpt-image-1.5` の `high` で画像を生成する。
* ベース画像（`lily_desktop/sys/lily_images/lily_character_sheet.png`）をキャラクターシートとして参照して生成する。
* 生成画像は美少女ゲームの様に、腰から上で生成する。
* 生成画像の背景は透過とする。
* 生成した画像は `lily_images/` に保存し、対応関係ファイルに自動追記する。
* これにより初回は生成が必要だが、最終的に全カテゴリの全バリエーションが揃う。
* ドキュメント: https://developers.openai.com/api/docs/models/gpt-image-1.5

### 8-5. 相方のポーズ管理
* 相方（葉留佳）は画像生成せず、既存の29枚のみで対応する。
* 既存画像を共通カテゴリにマッピングする（haruka_pose_map.json）。
* リリィ専用カテゴリが指定された場合は、最も近い共通カテゴリにフォールバックする。
* 該当カテゴリに複数枚ある場合は、ランダムまたは順番に1枚を選択する。

## 9. 相方システム
* 相方の画像は `lily_desktop/sys/aikata_images` に保存されている。
* 相方についても、立ち絵のファイル名と利用シーンの対応関係を示したファイルを持つ。
* 相方のセリフ内容に応じて立ち絵を変える。
* 会話中は、どちらが話しているか分かるように話者ごとの吹き出しを表示する。
* 相方のキャラクター設定(名前、性格など)はファイルで管理する。

## 10. カメラシステム
* リリィは設定しているカメラで外の様子を確認できる。
* 使用するカメラはトレイアイコンのサブメニューから選択できる。選択結果はconfig.yamlに保存する。
* 3分おきにカメラ画像を取得し、画像をAIで分析して外で何が起きているか把握する。
* カメラ画像解析では、人物が写っている場合は「人が何をしているか」を要約の中心にし、部屋や周囲の様子は補足として扱う。人物が写っていない場合は、その旨を明示したうえで部屋や周囲の様子を要約する。
* カメラ画像解析AIは provider を切り替え可能にする。`camera.analysis_provider` が `openai` のときは OpenAI Chat Completions を使い、`ollama` のときは Ollama `/api/chat` を使う。
* この仕様更新時点の既定値は `camera.analysis_provider: ollama` / `camera.analysis_model: gemma4:e4b` / `camera.analysis_base_url: http://127.0.0.1:11434` とする。
* カメラ画像はAI解析へ送る前に縮小しない。取得したPNGをそのまま解析AIへ送る。
* 同時に、デスクトップ状況の取得、現在アクティブなアプリの取得も行う。
* それらを時刻とともにローカルに記録する。
* サーバーには、30分おきに、その30分間の出来事を要約して記録する。30分要約AIも provider を切り替え可能にする。`camera.summary_provider` が `openai` のときは OpenAI Chat Completions を使い、`ollama` のときは Ollama `/api/chat` を使う。
* この仕様更新時点の既定値は `camera.summary_provider: ollama` / `camera.summary_model: gemma4:e4b` / `camera.summary_base_url: http://127.0.0.1:11434` とする。
* スクリーンショット解析AIも provider を切り替え可能にする。`desktop.analysis_provider` が `openai` のときは OpenAI Chat Completions を使い、`ollama` のときは Ollama `/api/chat` を使う。
* この仕様更新時点の既定値は `desktop.analysis_provider: ollama` / `desktop.analysis_model: gemma4:e4b` / `desktop.analysis_base_url: http://127.0.0.1:11434` とする。
* スクリーンショット解析で Ollama を使う場合は、構造化された最終回答を優先するため `think=false` で呼び出し、応答上限は `num_predict=500` とする。
* `desktop.*` が未設定の既存設定ファイルでは、従来どおり `openai.screen_analysis_model` を読み取り fallback として使う。
* desktopアプリ、webアプリ共にtool searchでその情報を使えるようにする
* その情報をもとに、リリィがユーザーへ話しかけることがある。相方も乗っかってくる。
* 自動雑談の話題選択は、デスクトップ状況・カメラ状況・Wikimedia（注目記事）・Wikimedia（興味分野）・Annict・健康豆知識・本・思い出・クエスト今日状況・クエスト週次状況 の10カテゴリを均等（各約10%）に配分する。カテゴリを先に抽選し、当たったカテゴリだけ実データを取得する。データが取得できないカテゴリは抽選から除外し残りで再配分する。

## 12. 技術
* PySide6で実装する。
* 会話AIは `gpt-5.4` を使う。設定で変更できるようにする。
* リリィデスクトップ本体の実行ログは JST で出力し、`lily_desktop/logs/runtime/YYYY-MM-DD.log` に日別で保存する。標準出力と同じ内容をファイルにも残す。

## 13. Health Planet 連携
* 時刻の基準は JST とする。
* Health Planet の同期は起動時に 1 回実行し、起動後は `lily_desktop/config.yaml` の `healthplanet.sync_interval_minutes` 間隔で再同期する。
* `healthplanet.sync_interval_minutes` の既定値は 15 分とし、0 以下や不正値は 15 分にフォールバックする。
* 同一計測の重複判定は `date` + `time` で行い、`lily_desktop/logs/health/YYYY-MM-DD.jsonl` に未保存の計測だけを新規として扱う。
* 新規計測が見つかった場合は、JST 上で最新 1 件だけを対象に `体重計測クエストクリア` をユーザー発話としてデスクトップのリリィへ送る。
* Health Planet 同期処理からは `complete_quest` を直接呼ばず、既存の会話フローでクエスト完了判定・吹き出し表示・会話保存・応答生成・読み上げを行う。
* 起動時にトークンが無効なら OAuth ダイアログを表示する。定期同期ではトークン無効時に OAuth ダイアログを連打せず、再認証待ちとして同期をスキップする。

## 14. Local HTTP Bridge
* リリィデスクトップは、同一PC上のローカルクライアント向けに Local HTTP Bridge を持つ。
* 待ち受け先は `127.0.0.1` 固定とし、v1 では認証を設けない。
* v1 の公開 endpoint は `POST /v1/events` のみとする。
* v1 の対応イベントは `user_message`、`system_message`、`quest_completed`、`chrome_audible_tabs` の 4 種とする。
* `user_message` / `system_message` / `quest_completed` は既存の会話入口へ流し、会話保存・吹き出し・応答生成・読み上げを既存フローで処理する。
* `system_message` はユーザー発話ではなく、保存 role は `system`、吹き出しはユーザー吹き出しと同位置の別色表示にする。
* `chrome_audible_tabs` は会話入口へは流さず、定期自動雑談スキップ判定用の直近 Chrome 可聴タブ状態として保持する。
* `source` と `metadata` はログに残すが、内部発話文には混ぜない。
* v1 では Quest / Completion の直接永続化は行わない。
* 詳細仕様は `spec/リリィデスクトップLocal_HTTP_Bridge仕様.md` を正本とする。

## 15. 内部イベント駆動ジョブ層
* UI 側の `core/event_bus.py` は従来どおり UI signal bus として維持する。
* 背景処理は別レイヤーの internal event/job layer で扱う。
* typed event は `core/domain_events.py`、job policy は `core/job_manager.py`、handler 登録は `core/background_event_runtime.py` に集約する。
* `ChatAutoTalkDue` のうち定期タイマー起点 (`source="auto_conversation.timer"`) は、通常会話中なら `main.py` 側で最新1件だけ保留し、5分以内なら会話終了後に投入する。
* 正本仕様は `spec/リリィデスクトップ_イベント駆動ジョブ設計.md` とする。

## 16. 冗長処理削減
* Health Planet 同期後にクラウド API へ再送するのは、その同期で新規に取得できた `new_records` のみとする。
* `new_records` が 0 件のときはクラウド側の `POST /health-data` を呼ばない。
* Tool Search の `chat_sessions` / `chat_messages` で JST 日付指定をした場合は `/chat-messages?from=...&to=...` を 1 回だけ呼び、候補 session ごとの N+1 取得は行わない。
* `sessionId` 指定の詳細取得のみ従来どおり `/chat-sessions/{id}/messages` を使う。
* Desktop の会話コンテキスト取得は user / skills / quests / completions / activity logs を 30 秒だけキャッシュし、クエスト mutation 成功時に invalidate する。
* `complete_quest` のスキル解決は Web 準拠の 2 段構えにする。`fixedSkillId` / `defaultSkillId` で即時解決できない completion は、OpenAI 利用可能時に `skillResolutionStatus = pending` で先に保存し、その後に非同期で再判定する。
* 非同期再判定の結果は `PUT /completions/{id}` で `resolved` / `needs_confirmation` / `unclassified` として反映し、必要に応じて `resolvedSkillId` / `skillXpAwarded` / `candidateSkillIds` / `resolutionReason` を更新する。
* 非同期再判定が `resolved` で、かつ quest の `skillMappingMode != ask_each_time` の場合は `PUT /quests/{id}` で `defaultSkillId` も更新する。
* OpenAI の API キー未設定、`privacyMode = no_ai`、通信失敗時はローカル fallback 判定で最終状態まで確定し、pending completion を残さない。
